GOOP 3 Class Template

The By Reference Model

The GOOP 3 class architecture adds a by-reference programming model to a standard LabVIEW class. This means that the class wire will not contain an object, instead it will contain a reference to the object. This programming model is the same as is used by for example LabVIEW Queue VI:s, LabVIEW Config File VI:s and many more.


The by reference model allows you to access the same object from separate loops or VI:s executing in parallel and to control when objects gets created and destroyed.


The GOOP 3 architecture

The new GOOP 3 architecture is built on the LV native class.

A GOOP 3 class is:

- A native class with an attribute holding the reference to the GOOP object.

- The GOOP ref is a data log file refnum with an enum put inside.

- GOOP object and class data are defined by two CTL:s; object Attributes.ctl and

class Attributes.ctl.


Look and feel

When programming with the GOOP classes they are in all essence native classes with by reference data. The GOOP object reference is not for public use and is not wired between method VI:s. The GOOP 3 class can of course have by value (native) attributes. The Create/Destroy methods supports initialization and cleaning up of native attributes as well as GOOP attributes.


File and project structure

File structure when generating code:

- A separate folder for each class. "Protected"/"Private" folders are for

VI:s/CTL:s with the corresponding visibility. Sub folder “util” is for GOOP 3 specific stuff.


Project structure for a class:

- A native lvclass with three project folders: Protected, Private and Util (used

to hide GOOP stuff). Protected and Private folders have the corresponding scope. VI:s of the Util folder have Protected or Private scope.  Private and protected are meant to be used to structure user defined protected/private class member VI:s and CTL:s. Public methods are placed directly below the lvclass node.


Create and Destroy

The Create and Destroy methods are automatically created by the tool. These methods are used to create and destroy objects of the class. The Destroy method is a dynamic dispatch VI and must therefore have the same VI name for all overrides. The Create method is not Dynamic Dispatch and is therefore not allowed to have the same VI name within an inheritance hierarchy. The reason for making Create non dynamic dispatch is to allow for different controls (for initialization) throughout the inheritance hierarchy. Destroy on the other hand should be dynamic dispatch to allow an object to be destroyed without checking its type. By convention Destroy does not take any other inputs than the object reference and error in since it must not depend on any parameters for proper object destruction.


Data of the class

GOOP 3 refers to the data fields of the class as “attributes”.


GOOP 3 supports object attributes and class attributes. The first are the ordinary data which will be unique for each object. The latter is data shared between all objects of a class (a.k.a static class members in C++).


As opposed to the standard lvclass the GOOP 3 attributes are protected which means the parent class data can be accessed from child classes without using get and set methods. This convention was used in GOOP 2 and has been preserved.


Class storage conventions and VI naming

Why does the tool impose a class storage convention?


Although LabVIEW allows the files of a class to be distributed in any way in the file system this is in practise hardly a good idea.


Placing all class VI:s and CTL:s in the same folder will not work if inheritance is used since dynamic dispatch overrides must have the same VI name and thereby cannot be placed in the same folder.


For GOOP 3 the lvclass concept has allowed us to remove the class name which in GOOP 2 and 1 was used as a prefix in all VI:s of a class. This is another reason why GOOP 3 classes cannot be placed in the same folder.


A recursive folder structure (class folders containing other class folders) should probably work but is not really intended or tested.
