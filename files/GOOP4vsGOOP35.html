GOOP 4 vs GOOP 3

GOOP4 vs. GOOP3

This section highlights differences between GOOP 4 and GOOP 3 and is intended for comparison, especially for existing GOOP 3 users. To get a more complete description of GOOP 4 we refer to the GOOP 4 class template section.


The object kernel

GOOP 4 has a new object kernel. Whereas GOOP 3 classes stores the by reference object data in an un-initialized shift register the new GOOP 4 classes uses the native DVR (Data Value Reference). Since the DVR is used there is very little extra code in a GOOP 4 class as compared to a standard LabVIEW class or a GOOP 3 class.


Each GOOP 4 class has a CTL file that defines a cluster of the by reference data of the class. This looks just like GOOP 3 and previous GOOP versions. The difference is that we use DVR to get a reference to the cluster data, i.e. the by reference object.


The object reference

As in GOOP 3 the reference is embedded into the native LabVIEW class. So a GOOP 4 class is a native LabVIEW class with one data field, “ref”, which contains the DVR reference. Therefore, in the code you cannot really see the difference between using a GOOP 4 by reference class or a standard by value class or a GOOP 3 class.


Access parent class data

In GOOP 3 the parent class data is accessed through unbundling the class data cluster after reading object data using the GetAttributes VI.


In GOOP 4 the data is accessed through a separate DVR reference on each class in the inheritance hierarchy. To access parent class data simply call the parent class GetAttributes VI to get the parent class data DVR, use this to read or modify parent class data.


Another way to access parent class data in GOOP 3 and GOOP 4 is to create protected/public methods that reads/writes data. This is often a more controlled way of accessing the data.


GOOP 4 – a light weight class template

The GOOP 4 class has all features of the GOOP 3 class and even more. But they are not added by default. The reason for this is that many of those features are not used in each class, therefore they often add dead code and complexity to the class. Instead these features are added explicitly to each class when they are needed. This is done using the Design Pattern concept. This concept enables you to add features like “persistent”, “active object”, “interface”, “class attribute” and “debugging” to an existing class.


Summary of benefits of GOOP 4 as compared to GOOP 3

  • Better run-time performance for read/write data thanks to usage of LabVIEW DVR.
  • Less generated code since the kernel VIs are replaced with the DVR. The extra generated code is kept to a minimum.
  • Less generated code by default since the standard class doesn’t have all features, only the most important ones. This way unused code is avoided. Features are added using the design pattern concept.
  • Less tool dependency since the generated code is less complex and kept to a minimum.
  • The Design Pattern concept that allows you to add features like “persistent”, “active object”, “interface”, “class attribute” and “debugging” to an existing class when they are needed.
