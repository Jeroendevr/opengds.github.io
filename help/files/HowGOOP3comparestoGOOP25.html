How GOOP 3 compares to GOOP 2

The (lack of) Wizard

There are some differences between GOOP 3 and GOOP 2.

There is no GOOP Wizard in GOOP3. Look under Finding the Features to see where they can be found.


Portability

In LabVIEW 8.2.1 lvclasses do no run on LV realtime. GOOP 2 classes runs on LV realtime and therefore enables an object oriented design on this target. Use the GOOP Inheritance Toolkit to create GOOP 2 classes.


Virtual Methods

In GOOP 3 the GOOP 2 virtual methods are replaced by LabVIEW Dynamic Dispatch VI:s. The dynamic dispatch VI:s has a constant lookup overhead independent on the depth of the inheritance hierarchy. The GOOP 2 virtual methods are run by VI server and dynamically loaded into memory at object creation (and thereafter they are cashed). The GOOP 2 virtual methods are likely to be slower than dynamic dispatch VI:s although it is not bench-marked. On the other hand they allow child classes to be used as dynamic plug-ins.


Maintaining the dynamic dispatch VI:s promises to be easier compared to the GOOP 2 virtual methods. The reason is that you don't need to wire up the Call By Reference Node call when you add/remove controls. Also, if there is a mismatch preventing execution the LabVIEW error window should pinpoint this.


Attributes

Reading/modifying object attributes in GOOP 3 should have the same performance as

GOOP 2 since the object repository of the GOOP kernel is not changed. This also warrants for a good stability of the GOOP 3 architecture. Thousands of classes have been created using the GOOP 2 class templates.


Access control

The  visibility of attributes and methods could not be enforced in GOOP 2. It was a convention using a different color for private VI icons and a private file folder. In GOOP 3 the visibility of methods and attributes is enforced by LabVIEW which is a great improvement.


The Object Reference

In GOOP 2 the object reference has the type of the root base class. The reason for this was that we didn't have any elegant way of sub typing the object reference. In GOOP 3 each class in the inheritance hierarchy will have its own object type. There are two benefits of this:


1. If you have two child classes you cannot wire an object from one class into the other

without getting a broken wire. In GOOP 2 you could wire up this scenario and instead get

a run-time error.


2. If you have several levels in an inheritance hierarchy the object reference may not be descriptive enough and you will probably use many objects from some intermediate base class rather than through the root. The most extreme example of this is the LabVIEW classes which has "Object" as root base class. If you implemented that in GOOP 2 all objects would share the same object reference, it would be easy to mix (mess) objects of different types.
